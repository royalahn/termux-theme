강의 핵심 포인트 4가지

우리가 놓치고 있었던 에이전트 활용의 정석

01. 하네스와 환경 설정

프롬프트를 매번 입력하지 마세요. CLAUDE.md와 AGENTS.md를 사용하여 프로젝트 룰, 기술 스택, 금지 사항을 환경에 한 번에 정의합니다. 이는 에이전트에게 신입사원 온보딩을 해주는 것과 같습니다.

  • 기술 스택 버전까지 명시 (예: Next.js 14 App Router)
  • 부정 룰("~하지 말 것")을 강력하게 적용

02. 멀티 에이전트 협업

하나의 에이전트가 기획부터 검증까지 모두 할 수 없습니다. Planner, Coder, Reviewer, Tester로 역할을 분리하여 컨텍스트 폭발을 막고 병렬로 실행하여 생산성을 극대화합니다.

  • 각 Sub-agent는 독립된 컨텍스트 윈도우 사용
  • 방향성 검증을 위해 /plan 모드 적극 활용

03. 강력한 플러그인 생태계

표준 셋업을 제공하는 Superpowers와 gstack 플러그인을 활용합니다. 개발 워크플로우를 강제로 TDD 사이클에 맞추거나, 가상의 팀 페르소나를 부여할 수 있습니다.

  • Superpowers: Hard Gate TDD 자동 발동
  • gstack: 6가지 강제 질문으로 완벽한 스펙 정의

04. RAG 기반 컨텍스트 관리

검색 증강 생성(RAG)을 통해 코드 에이전트의 검색 인프라를 이해합니다. 원본 문서를 청킹(Chunking)하고 벡터 DB에 저장하여 유사도 높은 컨텍스트를 빠르고 정확하게 제공합니다.

  • 모델의 한계(환각)를 보완하는 핵심 기술
  • 파일 횡단 및 코드 탐색의 효율성 향상

Part 0 & 1

AI 기초와 에이전트의 차이

  • 코드 어시스턴트 vs 에이전트: 기존에는 AI에게 질문하고 코드를 복사해 사람이 직접 실행했다면(어시스턴트), 에이전트는 목표를 받으면 파일 작성, 명령어 실행, 검증까지 스스로 자율 루프(이해→계획→실행→검증)를 돌며 작업합니다.
  • RAG (검색 증강 생성): 모델의 환각(거짓말)을 줄이기 위해 원본 문서를 쪼개고(청킹) 벡터 DB에 저장하여, 질문과 유사도 높은 컨텍스트를 주입하는 인프라입니다.
Part 2

하네스 (Harness): 환경에 한 번에 정의하기

AI에게 매번 코딩 스타일이나 스택을 설명할 필요가 없습니다. 신입사원 온보딩 매뉴얼처럼 CLAUDE.md 또는 AGENTS.md 파일 하나에 룰을 정의합니다.

  • 구체적 명시: "Next.js 14 App Router 사용", "Vitest로 단위 테스트" 등 버전까지 정확히 명시합니다.
  • 하지 말 것: 긍정 룰보다 부정 룰("any 타입 금지", "인라인 색상 금지")이 모델을 통제하는 데 훨씬 강력합니다.
Part 3

플러그인: Superpowers & gstack

  • Superpowers: 개발 방법론을 강제합니다. TDD(테스트 주도 개발)를 강제 발동하여 테스트가 통과(GREEN)해야만 다음 코딩으로 넘어가는 등 안정적인 개발을 돕습니다.
  • gstack: CEO, EM(엔지니어링 매니저), 디자이너 등 가상의 팀 페르소나를 부여하여 6가지 강제 질문(Forcing Questions)으로 개발 전 확실한 설계를 유도합니다.
Part 4 & 5

팀처럼 일하기 & 실전 풀스택 구축

  • 역할 분담: 한 에이전트가 모든 것을 하면 컨텍스트가 엉킵니다. 서브 에이전트(Sub-agents)를 활용하여 역할을 쪼개고 독립된 환경에서 병렬로 코드를 작성합니다.
  • Plan Mode 활용: 코드를 수정하기 전 반드시 읽기 전용인 /plan 모드로 방향성을 검토합니다. 실패 비용이 사실상 0원에 가깝습니다.
  • 최종 배포: 로컬 환경에서의 확인과 테스트가 끝난 후, Github Push를 통해 Vercel에 연동된 자동 배포 환경을 구축하는 실습을 진행합니다.

자주 빠지는 함정 3가지

  • Auto Mode 무인 실행 금지: 자율 루프가 통제 없이 폭주하면 크레딧이 전소될 수 있습니다. 단계마다 사람의 승인을 거치세요.
  • 오버 엔지니어링: 단순한 1줄짜리 버그 수정에 5명의 멀티 에이전트를 부르지 마세요. 가벼운 건 직접 고치는 게 훨씬 빠릅니다.
  • 사람의 리뷰 생략: 프로덕션 장애의 99%는 사람이 마지막 승인 게이트를 무시했을 때 발생합니다.

프롬프트 작성 4원칙

  • 목표를 동사로 명확히: "게시판 UI" 대신 "게시판 UI를 만들고 글쓰기가 되는지 확인해" 라고 지시하세요.
  • 컨텍스트 우선 제공: 룰, 참조 파일, 관련 선례를 가장 먼저 입력하세요.
  • 검증 방식 포함: "이 테스트 코드를 돌려서 확인해" 처럼 명확한 검증법을 제시하세요.
  • Plan Mode 우선: 냅다 코딩시키지 말고 /plan 으로 방향부터 점검하세요.

# Part 0~1 : 에이전트란?
  • 그동안 난 AI를 단순 '코드 복사기'로만 썼다 🤦‍♂️
  • 에이전트는 터미널 명령어도 치고 파일도 직접 수정하는 '자율 직원'이다.
  • 핵심: 어떻게 할지 하나하나 지시하지 말고, 명확한 '목표'만 던져줄 것!
# Part 2 : 하네스 (CLAUDE.md)
  • 매번 새 채팅 열 때마다 "이거 Next.js고 Tailwind 써줘" 타이핑하는 건 엄청난 낭비.
  • 프로젝트 루트에 CLAUDE.md 하나 파서 스택, 명령어, 금지사항 적어두면 매번 자동 적용됨!
  • 프롬프트 캐싱 덕분에 API 비용도 90% 아낄 수 있다 💰
# Part 3 : 플러그인 생태계
  • Superpowers: 코드 짜기 전에 무조건 '테스트 코드' 먼저 짜게 강제함 (TDD). 안정성 최고.
  • gstack: /office-hours 치면 가상의 CEO가 등장해서 "이거 왜 만듦?" 하고 기획 방향 날카롭게 잡아줌.
# Part 4~5 : 실전 협업 꿀팁
  • 에이전트 하나한테 다 시키면 뻗는다. 프론트/백엔드 서브 에이전트로 쪼개서 병렬로 일 시키기.
  • ⭐ 제일 중요 ⭐ 무작정 코드 짜게 하지 말고, 무조건 /plan 모드부터 실행할 것!
  • 읽기 전용이라 돈도 거의 안 들고, 기획서부터 검토해서 헛발질(환각)을 방지할 수 있음.
