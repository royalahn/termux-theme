1. MCP (AI 도구 연결)

MCP(Model Context Protocol)는 AI에게 새로운 능력을 부여하는 '플러그인'입니다. AI가 직접 웹을 검색하거나 웹사이트를 조작할 수 있도록 도구를 쥐어줍니다.

  • Context7: 프로젝트의 전반적인 맥락과 상황을 이해하는 도구입니다.
  • Firecrawl: 외부 문서를 검색하고 최신 정보를 읽어오는 도구입니다.
  • Playwright: 실제 브라우저를 띄워서 페이지를 테스트하는 도구입니다.

2. 서브 에이전트: Researcher

메인 AI를 도와주는 '자료조사 전용 비서'입니다. 코드를 수정하지 않고 오직 정보 수집과 사실 확인만 담당하여 시스템을 안전하게 유지합니다.

  • Firecrawl MCP 도구 하나만 사용해서 웹 검색을 전담합니다.
  • 코드를 실수로 망가뜨릴 위험이 없도록 읽기 전용(Read-only) 모드로 동작합니다.

3. 서브 에이전트: UI Tester

사람 대신 웹사이트에 접속해 클릭해보고 확인하는 'QA(품질보증) 비서'입니다. 만든 코드가 브라우저에서 제대로 작동하는지 검사합니다.

  • Playwright MCP를 사용하여 눈에 보이지 않는 서버를 띄운 뒤 페이지를 로드합니다.
  • 버튼 클릭, 모바일 화면 비율 확인, 네트워크 에러 유무 등 사람처럼 꼼꼼하게 점검합니다.

4. 자동화 워크플로우: Skills

'스킬(Skill)'은 AI가 여러 단계를 알아서 척척 처리하게 만든 업무 순서도입니다. 단 한 번의 명령으로 수정부터 테스트, 저장까지 이어집니다.

  • 사용자가 수정할 부분을 알려주면(예: 추천 일정 추가) 딱 그 부분만 수정합니다.
  • 수정이 끝나면 곧바로 UI Tester 비서를 호출하여 결과물이 정상인지 평가받습니다.
  • 테스트에 성공(PASS)하면 최종적으로 코드를 자동 저장(Commit)합니다.

```toml
검증 대상 선택 규칙:
1. repo root의 `index.html` 절대경로를 확인한다.
2. 먼저 file URL 전략을 고려한다.
   - Windows 예: `C:\\Users\\...\\index.html` -> `file:///C:/Users/.../index.html`
3. 브라우저 도구가 `file://` 직접 로드를 막거나 file URL이 정책상 실패하면 이를 UI 실패로 보지 않는다.
4. file URL이 실패하면 repo root에서 임시 localhost static server를 띄워 검증한다.
5. localhost URL 사용 전에는 반드시 서버가 실제 HTTP 응답을 하는지 확인한다.
6. `127.0.0.1:4173`이 응답하지 않으면 무작정 접속하지 않는다.
7. 기본 포트는 `4173`이고, 사용 중이면 `4174~4199` 중 사용 가능한 포트를 고른다.
8. static server는 아래 순서로 시도한다.
   - Windows: `py -3 -m http.server <port> --bind 127.0.0.1`
   - 공통: `python -m http.server <port> --bind 127.0.0.1`
   - 공통 대안: `python3 -m http.server <port> --bind 127.0.0.1`
   - Python이 없고 Node.js가 이미 있으면 `node -e` 기반 임시 static server를 사용한다.
9. Python/Node 모두 없으면 UI 실패가 아니라 `BLOCKED: no local static server runtime available`로 보고한다.
10. `npx serve`, `npm install`, `npm run dev`는 사용하지 않는다.
11. 테스트가 끝나면 subagent가 시작한 static server process를 종료한다.
12. `ERR_CONNECTION_REFUSED`는 UI 실패가 아니라 서버 미실행/잘못된 URL 선택 문제로 보고한다.

# Playwright MCP 검증 흐름
1. 선택한 URL로 `browser_navigate`를 수행한다.
2. 로드 직후 `browser_snapshot`으로 접근성 트리를 확인한다.
3. body blank 여부를 확인한다.
4. 주요 텍스트, heading, nav, hero, card, button, footer가 렌더링되는지 확인한다.
5. 클릭 가능한 요소는 snapshot의 `[ref=...]`를 확보한 뒤 `browser_click`으로 실제 클릭한다.
6. hover가 의미 있는 요소는 snapshot ref 기반으로 `browser_hover`를 수행한다.
7. 탭, 필터, 카드, 상세 패널, 모달, drawer, accordion 등 변경 요구사항에 포함된 인터랙션은 최소 1회 이상 직접 조작한다.
8. 조작 후 다시 `browser_snapshot`을 호출해 UI 상태 변화가 실제로 반영됐는지 확인한다.
9. `browser_resize`로 desktop/mobile 반응형을 확인한다.
   - desktop 예: 1440x900
   - mobile 예: 390x844
10. `browser_console_messages`로 콘솔 에러를 확인한다.
11. `browser_network_requests`로 404/500 등 네트워크 에러를 확인한다.
12. 의도한 변경 요구사항과 실제 UI 결과가 일치하는지 판단한다.
```

SKILL.md 본문에 아래 내용을 작성하라.

```markdown
# index-html-step-enhancer

## 목적

단일 `index.html`에서 사용자가 요청한 step 하나만 개선한다.

이 Skill은 직접 테스트 hook이나 git commit을 실행하지 않는다.  
`index.html` 수정 후 기존 Codex hook 흐름이 자동으로 이어지게 한다.

연결되는 기존 hook 흐름:

1. `index.html` 수정
2. 기존 `PostToolUse` hook이 HTML 기본 검증 수행
3. 필요 시 `playwright_ui_tester`로 실제 브라우저 검증 수행
4. 작업 종료 시 기존 `Stop` hook이 최종 테스트 및 자동 commit 수행

## step 매핑

- step 1: 도시 탐색 / `#cities`
- step 2: 7일 추천 일정 / `#itinerary`
- step 3: 예산 가이드 / `#budget`
- step 4: 꼭 알아야 할 여행 팁 / `#tips`
- step 5: 자주 묻는 질문 / `#faq`

## 실행 규칙

1. repo root와 `index.html` 존재 여부를 확인한다.
2. 사용자가 요청한 step 번호를 확인한다.
3. step 번호에 대응되는 section만 수정한다.
4. 요청되지 않은 section은 수정하지 않는다.
5. 한 번에 여러 step을 수정하지 않는다.
6. 필요한 경우 카드, drawer, modal, accordion, 상세 패널 등 UI 상호작용을 추가한다.
7. 모바일 반응형, 빈 화면 방지, 기본 접근성을 고려한다.
8. 외부 최신 정보가 필요할 때만 `firecrawl-researcher`를 최대 1회 호출한다.
9. 수정 후 `playwright_ui_tester`를 호출해 실제 브라우저 테스트를 수행한다.
10. `playwright_ui_tester`가 PASS이면 수정된 `index.html`을 그대로 둔다.
11. `playwright_ui_tester`가 FAIL이면 한 번만 수정 후 재테스트한다.
12. 재테스트도 FAIL이면 더 이상 수정하지 않고 중단 보고한다.
13. git staging/commit은 직접 수행하지 않는다.
14. 최종 commit은 기존 `Stop` hook이 자동 처리하게 둔다.

## 기존 hook 연결 규칙

이 Skill은 아래 기존 hook이 존재한다고 가정한다.

- `.codex/hooks.json`
- `.codex/hooks/post-html-validation.ps1` 또는 `.codex/hooks/post-html-validation.sh`
- `.codex/hooks/stop-test-and-commit.ps1` 또는 `.codex/hooks/stop-test-and-commit.sh`

연결 원칙:

- `index.html`을 수정하면 `PostToolUse` hook이 자동 검증해야 한다.
- 작업이 끝나면 `Stop` hook이 자동 테스트와 자동 commit을 담당해야 한다.
- Skill은 hook 파일을 수정하지 않는다.
- Skill은 hook을 수동 실행하지 않는다.
- Skill은 hook log를 직접 조작하지 않는다.
- hook이 실행되지 않는 것 같으면 `/hooks` trust 상태 확인이 필요하다고 보고한다.

## Playwright 검증 기준

`playwright_ui_tester`는 다음을 확인해야 한다.

- `index.html` 로드 여부
- body blank 여부
- 수정된 section 렌더링
- 주요 버튼/카드/패널 클릭 동작
- 콘솔 에러
- 네트워크/404 에러
- desktop/mobile 반응형

file URL이 막히면 UI 실패로 보지 말고, repo root에서 임시 localhost static server를 띄워 검증한다.  
localhost 접속 전 서버가 실제 응답하는지 확인한다.  
테스트 종료 후 임시 서버는 종료한다.

## commit 연결 규칙

이 Skill은 직접 commit하지 않는다.

commit은 기존 `Stop` hook이 담당한다.

Skill은 commit이 안전하게 수행되도록 아래를 지킨다.

1. 수정 대상은 `index.html` 하나로 제한한다.
2. `.agents/skills`, `.codex`, `.env`, hook, config 파일은 수정하지 않는다.
3. `git push`를 절대 실행하지 않는다.
4. Playwright 검증 PASS 전에는 작업 완료로 보고하지 않는다.
5. Playwright 검증 FAIL이면 commit이 진행되지 않아야 한다고 보고한다.
6. Stop hook이 자동 commit하지 않으면 `/hooks` trust 또는 hook 활성화 상태를 확인하라고 보고한다.

예상 commit message는 기존 Stop hook 정책을 따른다.  
별도 commit message를 Skill에서 강제하지 않는다.

- secret 출력 금지

- 남은 문제
```
