Prompt Engineering

LLM 성능 향상 가이드

최고의 AI 결과를 이끌어내는
18가지 프롬프트 원칙

Google, OpenAI, Anthropic의 연구 결과와 "Principled Instructions Are All You Need" 논문에 기반한 가장 효과적이고 실용적인 프롬프트 엔지니어링 기법들을 소개합니다.

#1

페르소나 (Persona) 정의

AI에게 '누구'의 관점인지 역할을 부여하면 답변의 관점, 판단 기준, 어조를 좁혀 일관성을 크게 높일 수 있습니다.

단, 최신 모델(Claude Code 등)에서는 단순한 역할 부여를 넘어 역할 + 작업 목표 + 현재 컨텍스트 + 완료 기준을 함께 제시하는 것이 좋습니다.

GOOD 
너는 10년 차 프론트엔드 아키텍트다.
현재 프로젝트를 읽고 index.html의 UI/접근성/반응형 문제를 리뷰하라.
바로 수정하지 말고 문제 목록과 개선 우선순위를 먼저 제시하라.
BAD 
내가 첨부한 index.html를 프론트엔드 엔지니어처럼 봐줘.
개선점을 도출해.
#2

원샷/퓨샷 (One-Shot/Few-Shot)

AI에게 추상적인 지시 대신, 원하는 결과물의 패턴과 형식을 직접 '예시'로 보여주어 효율을 극대화합니다.

Zero-Shot(예시 없음)보다 1~3개의 예시를 주었을 때 업무 형식 일치도와 정확도가 비약적으로 상승합니다. 리뷰 방식이나 출력 형식 등을 구체적인 예시로 보여주세요.

GOOD 
너는 접근성 리뷰어다.
출력은 아래 형식으로 고정하라:
1. 전체 판정
2. 문제 목록 (심각도, 문제, 근거)
3. 개선 우선순위
4. 테스트 기준
BAD 
index.html 좀 봐줘.
전문가처럼 확인해주고 문제 있으면 알아서 고쳐줘.
#3

생각의 사슬 (Chain of Thought)

답을 바로 요구하지 않고 '단계별로 생각하라(Let\'s think step by step)'고 지시하여 논리적 정답률을 높입니다.

AI가 스스로 논리적 단계를 밟아가게 유도함으로써 환각(Hallucination) 현상을 줄이고, 복잡한 추론 문제를 효과적으로 해결합니다.

GOOD 
이 기획안의 실현 가능성을 다음 단계별로 분석하라:
1. 예산 측면
2. 일정 측면
3. 인력 측면
으로 나누어 분석한 뒤 종합 결론을 확정하라.
#4

청중 설정 (Audience Persona)

결과물을 읽을 대상(청중)을 명확히 지정하여 AI가 어조와 난이도를 알맞게 자동 조절하도록 합니다.

전문가, 어린이, 비개발자 등 대상을 지정하면, 출력의 눈높이와 사용하는 어휘의 수준이 정확해집니다.

GOOD 
첨부한 내 프로젝트를 개발자가 아닌 비개발자 투자자들에게 쉽고 상세하게 설명하는 자료를 만들어라.
어려운 언어를 피하고 표와 예시를 포함해라.
#5

출력 결과에 예시 포함 요청

단순한 설명에 그치지 않고 구체적인 사용 사례(Showcases)나 시나리오를 함께 생성하도록 지시합니다.

추상적인 개념 설명보다 구체적인 예시를 생성하게 하면 답변의 활용도와 정확도가 동시에 상승합니다.

GOOD 
내 프로젝트에 A 기법을 적용하고 싶다. 구상을 단계별로 작성하고, A 기법이 구현되었을 때 유저들의 사용 시나리오를 3가지 예시로 생성하라.
#6

역질문 (Reverse Questioning)

정보가 부족할 때 AI가 멋대로 추측하게 두지 말고, 사용자에게 직접 질문하여 정보를 보충하도록 지시합니다.

초기 프롬프트가 모호하거나 누락된 맥락이 있을 때 품질 저하를 막아주는 훌륭한 방어 기제입니다.

GOOD 
이 기획안을 보강하고 싶은데, 네가 더 좋은 답변을 내놓기 위해 나한테 더 알아야 할 정보가 있다면 질문 3가지를 먼저 해라.
#7

구체적인 제약 사항 추가

AI에게 자유도를 낮추고 적절한 제약을 걸어 더 깊고 실용적인 답변을 유도합니다.

단어 수, 특정 단어 포함/배제, 엄격한 형식 지정 등 제약을 추가할수록 출력의 품질 관리가 용이해집니다.

GOOD 
초등학생 저학년이 이해할 수 있는 쉬운 단어를 활용하여, 반드시 200자 이내로 요약해서 작성하라.
#8

출력물의 구조 먼저 정의

작성을 시작하기 전에 뼈대가 되는 구조를 지정해주면 논리적 흐름이 탄탄한 결과물을 얻을 수 있습니다.

표 형식, 글머리 기호, 혹은 특정 목차 순서 등을 명시하면 AI가 내용을 빠뜨리지 않고 채워 넣습니다.

GOOD 
답변은 반드시 다음 순서로 작성하라:
1. 개요
2. 문제점
3. 해결책
4. 기대효과
#9

반복과 강조

긴 프롬프트에서 가장 중요하거나 절대 잊지 말아야 할 명령은 끝부분에 한 번 더 강조합니다.

AI 모델은 프롬프트의 시작과 끝을 가장 잘 기억하는 경향이 있으므로, 핵심 지시사항을 마지막에 재차 언급하는 것이 좋습니다.

GOOD 
(앞선 긴 설명 후 마지막에)
기억해라: 반드시 박사(PhD) 수준의 깊이 있는 분석이어야 하며, 요약본은 제외해야 한다.
#10

프롬프트 경계 분리

XML 태그나 Markdown 구분자를 사용하여 지시사항, 데이터, 예시 등을 명확히 분리합니다.

복잡한 프롬프트에서 모델이 어디까지가 룰이고 어디부터가 입력 데이터인지 혼동하는 것을 막아줍니다.

GOOD 
<role>너는 접근성 리뷰어다.</role>
<context>단일 HTML 파일 품질을 평가한다.</context>
<task>UI 문제를 진단하라.</task>
#11

근거 기반 답변 강제 (Grounding)

모호하고 일반적인 평가가 아닌, 실제 코드나 문서에 존재하는 명확한 데이터를 바탕으로 답변하도록 지시합니다.

근거 없는 '좋아 보인다' 식의 무의미한 답변을 차단하고, 실질적이고 객관적인 피드백을 이끌어냅니다.

GOOD 
모든 판단은 반드시 실제 파일명, 코드 구조, 선택자명 등을 근거로 제시하라.
근거 없이 '개선 가능하다'와 같은 일반적 표현은 사용하지 마라.
#12

도구 사용 규칙 (Tool-use Policy)

코딩 에이전트에게 언제 읽고, 언제 코드를 수정하며, 언제 테스트할지 명확한 시점과 권한을 지시합니다.

AI가 독단적으로 코드를 망가뜨리는 것을 방지하고, 제안과 구현 단계를 사용자의 통제 하에 둡니다.

GOOD 
먼저 파일을 읽고 진단하라. 1차 응답에서는 절대 파일을 수정하지 마라.
내가 '수정 진행'이라고 말한 뒤에만 코드를 수정하라.
#13

작업 분해 (Least-to-Most)

크고 복잡한 작업을 한 번에 처리하지 말고, 논리적인 하위 단계로 잘게 쪼개어 순차적으로 해결하게 합니다.

설계, 구현, 검증이 뒤섞여 프로젝트가 엉키는 것을 막고, 각 단계별로 정확성을 확보할 수 있습니다.

GOOD 
이 작업을 한 번에 구현하지 마라. 다음 순서로 진행하라:
1. 구조 파악
2. 문제 분류
3. 수정 계획 수립
4. 승인 후 구현
#14

자가 검증 강제 (Self-Check)

AI가 최종 답변을 출력하기 전에, 주어진 체크리스트(루브릭)에 맞춰 스스로 내용을 검증하게 합니다.

지시사항 누락이나 논리적 오류를 AI 스스로 한 번 더 점검하여 걸러내는 훌륭한 디버깅 기법입니다.

GOOD 
최종 답변 전에 다음 기준으로 스스로 검증하라:
- 실제 파일을 근거로 했는가?
- 수정 금지 지시를 지켰는가?
누락되었다면 보완 후 답변하라.
#15

상태 및 컨텍스트 관리 지시

장기적인 프로젝트를 진행할 때 `progress.md` 등 파일에 현재 작업 상태와 남은 과제를 기록하게 합니다.

에이전트가 기억을 잃거나 세션이 재시작되더라도, 기록된 상태 파일을 읽고 작업을 매끄럽게 이어나갈 수 있습니다.

GOOD 
작업이 길어지면 `progress.md`를 만들어 완료 작업, 남은 작업, 수정한 파일 등을 기록하라.
새 작업을 시작할 때 항상 이 파일을 먼저 읽어라.
#16

하드코딩 방지 (General Solution)

AI가 특정 테스트 케이스만 꼼수로 통과하기 위해 하드코딩(임시방편)하는 것을 강력히 금지합니다.

다양한 상황과 유효한 입력 모두를 커버할 수 있는 근본적이고 '일반적인' 로직(General Solution)을 작성하도록 유도합니다.

GOOD 
테스트를 통과시키기 위한 꼼수나 하드코딩을 절대 하지 마라.
모든 유효한 입력 상황에서 동작하는 구조적이고 일반적인 해결책을 구현하라.
#17

다중 후보 생성 후 선택

중요한 설계 판단 시, 처음부터 하나의 답을 내지 단지 않고 여러 대안을 먼저 제시한 뒤 장단점을 비교하게 합니다.

최소 수정안, 디자인 중심안, 성능 중심안 등 다각도의 사고(Tree-of-Thoughts)를 거쳐 최적의 선택을 하도록 돕습니다.

GOOD 
바로 하나를 확정하지 마라. 개선 방향을 3가지 후보(A/B/C안)로 제시하고,
각 안의 장단점을 비교 분석한 뒤 가장 적합한 것을 추천하라.
#18

불확실성 및 Fallback 명시

작업 중 정보가 부족하거나 오류를 만났을 때, AI가 멋대로 추측하지 않고 어떻게 대응할지 행동 지침을 정해줍니다.

확인 불가함을 명시하거나 사용자에게 권한을 넘기는 등 예측 가능한 안전망(Fallback Policy)을 구축합니다.

GOOD 
정보가 부족하면 절대 추측하지 마라.
확인 불가능한 내용은 '확인 불가'로 명시하고, 판단 정보가 부족하면 즉시 질문하라.
